# GitHub publishing setup

The Site Manager publishes every customer site to one shared repository. Each
site gets one generated file:

```text
sites/<site-uuid>/site.config.json
```

The `publish-site-to-github` Edge Function creates the commit. Credentials are
kept in Supabase secrets and never sent to the browser.

## 1. Create the GitHub App

In GitHub, open **Settings > Developer settings > GitHub Apps > New GitHub App**.

Suggested values:

- GitHub App name: `Reef Sites Publisher` (must be globally unique)
- Homepage URL: the Site Manager production URL
- Webhook: inactive for this first publishing integration
- Repository permissions > Contents: **Read and write**
- All other repository and organization permissions: **No access**
- Where can this GitHub App be installed?: **Only on this account**

Create the app, record its **App ID**, and generate a private key. The downloaded
PEM file is a secret and must not be committed to this repository.

## 2. Install the app on the shared repository

From the GitHub App settings, select **Install App**. Choose **Only select
repositories** and select the one repository Netlify watches.

Record the numeric installation ID from the installation URL:

```text
https://github.com/settings/installations/12345678
                                        ^^^^^^^^
```

## 3. Add Supabase secrets

From `deriv-sites-ui-kit-main`, authenticate the Supabase CLI and run:

```sh
supabase secrets set GITHUB_APP_ID="your-app-id"
supabase secrets set GITHUB_APP_INSTALLATION_ID="your-installation-id"
supabase secrets set GITHUB_REPO_OWNER="your-user-or-organization"
supabase secrets set GITHUB_REPO_NAME="your-shared-repository"
supabase secrets set GITHUB_REPO_BRANCH="main"
supabase secrets set GITHUB_SITES_CONFIG_ROOT="sites"
supabase secrets set --env-file ./.env.github-publisher
```

Use the env file only for the multiline private key:

```env
GITHUB_APP_PRIVATE_KEY="-----BEGIN RSA PRIVATE KEY-----\n...private key contents...\n-----END RSA PRIVATE KEY-----"
```

Delete that local secrets file after setting it. The `.env.*` rule in the root
`.gitignore` prevents it from being committed accidentally.

## 4. Apply and deploy

```sh
supabase db push
supabase functions deploy publish-site-to-github
```

The function requires a signed-in Supabase user, verifies that the user owns the
site (or is a platform admin), creates a publish version and deployment record,
then commits the generated public configuration.

## 5. Connect Netlify once

The Site Manager can now provision a separate Netlify project for each customer
from the same shared repository. Create a Netlify personal access token owned by
the platform account and set these additional Supabase secrets:

```sh
supabase secrets set NETLIFY_ACCESS_TOKEN="your-netlify-personal-access-token"
supabase secrets set NETLIFY_ACCOUNT_SLUG="your-netlify-team-slug"
supabase secrets set NETLIFY_GITHUB_INSTALLATION_ID="netlify-github-app-installation-id"
supabase secrets set NETLIFY_WEBHOOK_SECRET="a-long-random-secret"
```

`NETLIFY_GITHUB_INSTALLATION_ID` is the installation ID of Netlify's GitHub App,
not the Reef Sites Publisher App. First connect the shared repository to one
sample Netlify project through the Netlify UI, then read the installation ID from
the configured Netlify GitHub App URL in GitHub.

Deploy all publishing functions:

```sh
supabase functions deploy publish-site-to-github
supabase functions deploy provision-site-netlify
supabase functions deploy netlify-deploy-webhook --no-verify-jwt
supabase functions deploy public-site-config --no-verify-jwt
```

The provisioning function creates a Netlify project with:

- base directory `new-user-interface-main`
- build command `npm run build`
- publish directory `dist`
- `REEF_SITE_ID=<site UUID>` build environment variable
- the customer's custom domain and forced SSL
- deploy-created, deploy-succeeded, and deploy-failed callbacks

The shared `netlify.toml` uses a site-aware ignore command. A configuration
commit builds only the affected customer's Netlify project. Changes to the
shared trading application or shared packages rebuild all customer projects.

The default commit message is generated by the server:

```text
chore(sites): publish <hostname-or-site-id> v<version>
```

## Offline behavior

The browser needs internet long enough for the Update Website request to reach
Supabase. After the server accepts it, GitHub authentication and committing are
server-side and do not require the tab to remain open. Deployment history is
stored in `site_deployments`, so the result is visible when the user returns.

Netlify callbacks extend deployment status from `committed` through `building`
to `deployed`, or to `build_failed`. Successful callbacks also update the
primary domain and SSL status.

## Operational order for a new customer site

1. Create the site and domain in Site Manager.
2. Configure branding, Deriv OAuth, pages, bots, and tools.
3. Create a draft and use Preview Draft.
4. Select Provision Website to create the Netlify project.
5. Select Update Website to generate and commit all site artifacts.
6. Follow progress in Recent updates and open the Netlify deploy log if needed.
7. Retry a failed update or restore a previously deployed version from the same panel.

Generated files are committed atomically in one commit:

```text
sites/<site-id>/site.config.json
sites/<site-id>/bots-manifest.json
sites/<site-id>/branding.css
```
